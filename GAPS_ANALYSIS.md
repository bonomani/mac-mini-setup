# Mac Mini AI Setup - Gaps Analysis

## Overview

This document categorizes identified gaps in the mac-mini-setup framework as either **Bug** (critical issues) or **New Feature** (enhancements deferred to future implementation).

Last updated: 2026-04-01

---

## Bugs

### Bug #1: Missing Verification Tests for AI Apps Services
**Status**: FIXED  
**Resolution**: HTTP endpoint probes added to `tic/software/verify.yaml` for all 5 Docker Compose services (Open WebUI, Flowise, OpenHands, n8n, Qdrant) with `skip_when` guards using `_tic_service_not_running`.

---

### Bug #2: Custom-Daemon Driver Lacks Action Implementation
**Status**: Not a Bug — Intentional Design  
**Resolution**: The `custom-daemon` driver intentionally implements only observe functionality. Ollama daemon lifecycle is managed externally (launchd/installer). The driver observes state (`running`/`stopped`/`absent`) and collects evidence.

---

### Bug #3: Compose-File Driver Lacks Install/Update Actions
**Status**: Not a Bug — Intentional Design  
**Resolution**: The `compose-file` driver intentionally implements only observe. The Docker Compose stack file is generated by `_ai_apply_compose_runtime()` in `lib/ai_apps.sh` via the sentinel-based runtime manager.

---

### Bug #4: TIC System Verification Depends on UCC Status File
**Type**: Design Limitation  
**Status**: Partially Fixed  
**Severity**: Medium  
**Component**: tic/system/verify.yaml, lib/tic_runner.sh

**Current State**: The oracle now uses the `_tic_target_status_is` wrapper instead of inline awk, but still depends on the UCC status file. No independent system health checks exist.

**Recommended Fix**: Add independent verification tests that don't rely on UCC status:
```yaml
# tic/system/verify.yaml (enhancement)
tests:
  - name: docker-desktop-running
    component: system
    intent: "Docker Desktop must be running to execute AI apps"
    oracle: 'docker_daemon_is_running'
    trace: "component:system / verification:docker-daemon"

  - name: ollama-api-reachable
    component: system
    intent: "Ollama API must be reachable for local LLM inference"
    oracle: 'http_probe_localhost 11434'
    trace: "component:system / verification:ollama-api"

  - name: python3-available
    component: system
    intent: "Python 3 must be available for AI stack"
    oracle: 'is_installed python3'
    trace: "component:system / verification:python-version"

  - name: homebrew-installed
    component: system
    intent: "Homebrew package manager must be installed"
    oracle: 'is_installed brew'
    trace: "component:system / verification:homebrew"
```

---

### Bug #5: ai-healthcheck Script Doesn't Validate Service Health
**Status**: FIXED  
**Resolution**: The `scripts/ai-healthcheck` script now includes HTTP health probes for all AI app services, Docker daemon status, Ollama API reachability, and a healthy/total summary with exit code.

---

### Bug #6: Missing Dependency Chain Verification Tests
**Type**: Feature Gap  
**Status**: Open — Low Severity  
**Component**: tic/software/verify.yaml

**Description**: No TIC tests verify that dependency chains are properly established. The framework's `depends_on` mechanism handles ordering at convergence time, but there is no post-convergence check that prerequisites remain satisfied.

**Note**: Any proposed oracle must comply with Rule 1 (no shell operators in YAML). Use existing lib functions like `is_installed`, `docker_daemon_is_running`, etc.
